# MetricsNode：对齐 Core v0.4.0 / SDK v0.1.4

## 变更背景 / 目标
- `MyFlowHub-Core v0.4.0` 与 `MyFlowHub-SDK v0.1.4` 已发布，本轮需要将 `MetricsNode` 主模块及相关子模块统一对齐。
- 本仓继续沿用 `main` + `debug-latest` 的发布方式，本次不引入新的 semver tag 策略。

目标：
- 将 `go.mod`、`nodemobile/go.mod`、`windows/go.mod` 中涉及的 `myflowhub-core` / `myflowhub-sdk` 版本统一升级。
- 在不改业务逻辑的前提下完成独立模块验证。

## 具体变更内容
- 依赖升级：
  - `go.mod`、`go.sum`
  - `nodemobile/go.mod`、`nodemobile/go.sum`
  - `windows/go.mod`、`windows/go.sum`
- 版本变更：
  - `github.com/yttydcs/myflowhub-core: v0.2.0 -> v0.4.0`
  - `github.com/yttydcs/myflowhub-sdk: v0.1.2 -> v0.1.4`
- 随依赖求解联动更新：
  - `golang.org/x/sys` 升级到 `v0.42.0`
  - `github.com/godbus/dbus/v5` 在主模块与子模块中提升到 `v5.2.2`

## 对应计划任务映射
- `REM-2`：MetricsNode 对齐 Core v0.4.0 / SDK v0.1.4

## 关键设计决策与权衡
- 保持发布习惯不变：不在本轮顺手引入 semver 发布体系，避免将“依赖对齐”扩大为“发布流程重构”。
- 子模块同步执行 `go mod tidy`：确保 `nodemobile`、`windows` 在 `GOWORK=off` 下可独立解析依赖，减少后续 CI / 本地构建偏差。
- `windows` 验证继续沿用既有占位策略：`frontend/dist` 属于被忽略的构建产物目录，测试前临时创建 `.keep` 即可满足 `go:embed`，不将前端生成物纳入版本库。

## 测试与验证方式 / 结果
在 `go1.25.0`、`GOWORK=off` 下执行：
- 仓库根：`go test ./... -count=1` ✅
- `nodemobile`：`go test ./... -count=1` ✅
- `windows`：
  1. 临时创建 `windows/frontend/dist/.keep`
  2. `go test ./... -count=1` ✅

说明：
- `windows/frontend/dist/.keep` 仅用于本地验证 `go:embed` 前置条件，目录本身已在 `.gitignore` 中忽略，未纳入提交。

## 潜在影响
- `MetricsNode` 现在默认依赖 `Core v0.4.0` 与 `SDK v0.1.4`，若下游仍固定旧版本，可能出现接口或依赖解析冲突。
- 构建 `windows` 子模块时，仍需保证 `frontend/dist` 至少存在一个文件；这是当前 Wails 结构的既有约束。

## 回滚方案
- 回退本次提交，并重新执行三个模块的 `go mod tidy` 以恢复旧依赖图。
- 若仅应用层发现兼容问题，优先追加最小修复提交，不直接回退整个版本对齐。
